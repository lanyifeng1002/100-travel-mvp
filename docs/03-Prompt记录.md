# Prompt 记录

说明：

以下 Prompt 不是为了展示“一句话生成项目”，而是按照工程化开发过程整理的阶段性 Prompt。它们基于本项目的真实开发步骤重构而来，适用于通用 AI 编程助手。

---

## 1. 需求拆解与架构设计阶段

**原始 Prompt**

```text
你现在是这个项目的架构师。我要做一个“轻量级旅行目的地展示 Web App”，目标不是预订平台，而是帮助用户浏览旅行灵感、筛选方向、查看详情，并能持续录入新内容。请先不要直接写代码，而是按“输入、处理流程、状态变化、输出、上下游影响”拆解这个项目，给出一个最小完整方案。要求优先保证项目能跑通、能演示、能扩展，不要引入登录、支付、评论等无关功能。最后把方案拆成可以顺序推进的小目标。
```

**说明**

这一步的目的不是让 AI 立刻生成页面，而是先收口范围，避免把项目误做成旅游社区或预订系统。最大的挑战是需求本身很容易被 AI 扩写，所以 Prompt 里必须明确“最小完整方案”和“非目标”。

---

## 2. 数据库定义与后端闭环阶段

**原始 Prompt**

```text
在现有前端展示基础上，继续把项目补成前后端闭环。请你新增 Express + SQLite，并满足以下要求：1）数据库只存目的地内容，不扩展到用户系统；2）需要有初始化脚本和重置脚本；3）提供列表、详情、标签、随机探索、相关推荐、统计信息接口；4）前端后续要能直接从真实接口读取。请先定义数据表字段，再实现后端入口、仓储层和数据库初始化逻辑。
```

**说明**

这一段 Prompt 把“后端怎么做”限制得非常具体，否则 AI 容易在数据库表设计上加太多冗余字段。这里的关键不是用什么技术，而是先让数据结构和接口边界稳定下来。

---

## 3. 核心业务逻辑实现阶段

**原始 Prompt**

```text
请继续完成前台页面，要求首页至少包含：目的地列表、关键词搜索、标签筛选、随机探索、空状态处理；详情页至少包含：完整介绍、亮点、适合人群、出行提醒、相关推荐。前端必须通过真实 API 获取数据，不再直接依赖本地写死数据。实现时优先保证浏览链路完整和交互清晰，不需要加入无关动画或复杂状态管理。
```

**说明**

这一阶段的重点是“从页面可看”变成“业务链路闭合”。最常见的问题是 AI 会只做静态页面，所以 Prompt 里必须强调“必须通过真实 API 获取数据”。

---

## 4. 内容管理与后台录入阶段

**原始 Prompt**

```text
请继续补一个最小可用的内容管理页，不需要登录系统，但需要有明确的录入表单和后台接口。表单字段必须覆盖 slug、名称、地区、图片、时间、预算、标签、亮点、适合人群和出行提醒。保存后要求前台列表和详情页能直接读取到新增内容。请同时补充后端的字段校验和重复 slug 处理。
```

**说明**

这一段 Prompt 是为了让项目从“展示型页面”变成“可维护的内容系统”。这里遇到的难点是如果不强调字段校验，AI 往往只会把表单做出来，但不会处理真实写入和错误提示。

---

## 5. UI/UX 细节调优阶段

**原始 Prompt**

```text
现在请从产品化视角检查整个项目，把所有“作业、课程、演示、MVP、假数据、后端接口、SQLite”这类工程说明式文案从公开页面去掉。公开页面应该更像一个正式的目的地内容产品，管理页则保留内部系统口径。不要大改结构，重点优化头部品牌、首页首屏、详情页提示语和管理页标题，让整体更自然、更像真实项目。
```

**说明**

这一步解决的是“项目能跑但看起来像作业”的问题。AI 在前期实现时容易留下很多技术自述式文案，所以后期必须用独立 Prompt 做一轮产品化收口。

---

## 6. Bug 修复与测试阶段

**原始 Prompt**

```text
请对当前项目执行一轮工程化收尾，不要只做表面检查。要求：1）补充自动化测试，至少覆盖种子数据数量、仓储层创建逻辑、核心 API；2）执行数据库重置、测试和构建；3）修复发现的问题并说明原因。特别注意检查路由兜底、数据库灌库数量和公开文案是否存在不一致。
```

**说明**

这一步用来逼 AI 进入“验证和修正”模式，而不是继续堆功能。本项目在这一阶段实际暴露出几个典型问题，比如 Express 5 的路由兜底写法、种子数据最初生成成了 115 条而不是 100 条，以及页面文案过于工程化。

---

## 7. 本项目中 AI 协同遇到的典型问题

### 问题 1：数据库灌库条数超出目标

现象：

- 目标是 100 条种子数据
- 初次生成后实际变成了 115 条

修正方式：

- 重新检查种子生成逻辑
- 收口基础目的地数量
- 执行 `npm run db:reset` 验证最终条数

### 问题 2：Express 5 的通配路由报错

现象：

- 直接使用传统 `app.get('*')` 兜底时出现 `PathError`

修正方式：

- 改成静态资源托管后的中间件兜底
- 先排除 `/api` 路径，再返回 `index.html`

### 问题 3：页面文案太像“技术演示”

现象：

- 首页和详情页出现了大量 `MVP`、`API`、`SQLite`、`作业` 等词

修正方式：

- 单独发起一次“产品化文案清洗”任务
- 将对外页面与内部管理页的口径拆开处理
